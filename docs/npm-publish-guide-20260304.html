How to Publish an npm Package v1.0

A complete guide to publishing @royrai/ui-kit (or any scoped package) to the npm registry

Created: March 4, 2026

This guide walks you through the entire process of publishing a package to the npm registry — from creating your npm account to making your first public release. By the end, anyone in the world will be able to install your package with a single npm install command.

We'll cover four main phases: setting up your npm account, preparing your package for publishing, performing the actual publish, and managing updates. The entire process takes approximately 15–20 minutes for the first publish.

Phase 1: Prerequisites

What you need before starting

Before you can publish, make sure you have the following tools installed and your package builds successfully.

1 Verify Node.js and npm

You need Node.js (which includes npm) installed on your machine. Check your versions:

node --version
npm --version

You need Node.js 18+ and npm 8+. If you don't have them, download from:

🔗
https://nodejs.org
Download the LTS version of Node.js
2 Verify your package builds

Run the typecheck and build commands to make sure everything compiles without errors:

yarn typecheck && yarn build

Both commands must pass with zero errors. If the build fails, fix the issues before proceeding — you cannot publish a broken package.

⚠️ Don't skip this

If you publish a package that doesn't build, everyone who installs it will get broken code. Always verify before publishing.

Phase 2: npm Account Setup

Create your account and organization on npmjs.com

npm needs to know who you are before you can publish. You'll create an account and (optionally) an organization for your @royrai scope.

1 Create an npm account

Go to the npm website and sign up for a free account:

🔗
https://www.npmjs.com/signup
npm sign up page — create a free account

Fill in your username, email, and password. You'll receive a verification email — click the link to activate your account.

2 Create an organization (for @scope)

Your package is named @royrai/ui-kit. The @royrai part is called a scope. To publish under a scope, you need an npm organization with that name.

🔗
https://www.npmjs.com/org/create
Create an npm organization
  1. Enter royrai as the organization name
  2. Select the free/unlimited plan (public packages only)
  3. Click Create
💡 Scope alternatives

If the org name royrai is taken, you can also use your personal username as a scope. For example, if your npm username is royrai, you can publish as @royrai/ui-kit without creating an organization.

⚖️ Free vs Paid npm plans

Feature Free Pro / Teams ($7/mo)
Public packages Unlimited Unlimited
Private packages Not available Unlimited
Team members Not available Unlimited

For publishing @royrai/ui-kit as a public package, the free plan is sufficient.

3 Log in from the terminal

Connect your local machine to your npm account so you can publish from the command line:

npm login

You'll be prompted for your username, password, and email. Once logged in, verify with:

npm whoami

This should print your npm username.

💡 Stay logged in

You only need to run npm login once per machine. npm stores your authentication token in ~/.npmrc and remembers it across sessions.

Phase 3: Prepare Your Package

Verify your package.json and build output before publishing

Before hitting publish, verify that your package is correctly configured. These checks ensure consumers get the right files and can import your components.

1 Check package.json essentials

Open your package.json and verify these critical fields:

What to check: These fields tell npm and consumers how to find and use your package. Getting them wrong means broken installs or missing files.
Key fields: name (must match your npm org/scope), version (semantic versioning), files (what gets uploaded), exports (import paths). 
{
  "name": "@royrai/ui-kit",
  "version": "0.1.02",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "files": ["dist", "README.md", "LICENSE"],
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./controls/RrBox": { "..." : "..." },
    "./controls/RrDataGrid": { "..." : "..." }
  }
}
Field Purpose
name The package name consumers will use in npm install
version Must be unique — npm rejects duplicate versions
files Only these files/folders are uploaded to npm (keeps the package small)
main Entry point for require() (CommonJS)
module Entry point for import (ESM)
types Entry point for TypeScript type definitions
exports Maps import paths to actual files (e.g., @royrai/ui-kit/controls/RrBox)
2 Preview what will be published

Before publishing, preview exactly which files npm will upload. This is a dry run — nothing gets published.

npm pack --dry-run

This prints a list of all files that would be included in the package. Verify that:

  1. The dist/ folder and all its contents are included
  2. README.md and LICENSE are included
  3. Source code (src/), config files (tsup.config.ts), and node_modules/ are NOT included
⚠️ Watch out for sensitive files

Never publish .env files, API keys, or credentials. The files field in your package.json acts as an allowlist — only listed files get published. Your .npmignore file provides additional protection.

3 Build one final time

Always build fresh right before publishing to ensure the dist/ folder matches your latest source code:

yarn typecheck && yarn build
💡 Automatic pre-publish check

Your package.json already has "prepublishOnly": "yarn typecheck && yarn build". This script runs automatically before every npm publish, so even if you forget to build manually, npm will do it for you.

Phase 4: Publish to npm

Push your package to the registry

This is the moment — one command makes your package available to the world.

1 Publish the package

From your project root directory, run:

npm publish --access public
Why --access public? Scoped packages (starting with @) default to private on npm. The --access public flag explicitly makes the package public and free. Without it, npm will reject the publish unless you have a paid plan.

npm will:

  1. Run your prepublishOnly script (typecheck + build)
  2. Pack the files listed in "files"
  3. Upload the tarball to the npm registry
  4. Print a success message with the package URL
✅ That's it!

Your package is now live. Anyone can install it with npm install @royrai/ui-kit

2 Verify on npmjs.com

After publishing, verify that your package appears on the npm website:

🔗
https://www.npmjs.com/package/@royrai/ui-kit
Your package page on npm (visible after publishing)

The page shows your README, version, dependencies, and download stats. It may take a minute to appear after the first publish.

3 Common errors and fixes
Error Cause Fix
E403 You don't have permission to publish under this scope Create the npm org first, or check that you're logged in as the right user
E402 Trying to publish a private scoped package on a free plan Add --access public to the publish command
EPUBLISHCONFLICT This exact version already exists on the registry Bump the version number in package.json
ENEEDAUTH Not logged in Run npm login first

Phase 5: Updates & Versioning

How to publish new versions of your package

After your first publish, you'll need to release updates. npm uses semantic versioning — each version number communicates the type of change to consumers.

1 Understand semantic versioning

Version numbers follow the format MAJOR.MINOR.PATCH:

Part When to bump Example
MAJOR Breaking changes — consumers must update their code 1.0.0 → 2.0.0
MINOR New features — backwards compatible 1.0.0 → 1.1.0
PATCH Bug fixes, style tweaks — no API changes 1.0.0 → 1.0.1
📌 Critical rule

npm does not allow publishing the same version twice. Every publish must have a unique version number. If you try to re-publish an existing version, npm will reject it.

2 Bump the version

You can bump the version manually in package.json, or use the npm command:

# For a bug fix:
npm version patch

# For a new feature:
npm version minor

# For a breaking change:
npm version major
What this does: The npm version command updates the version field in package.json, creates a git commit, and tags it. It's a shortcut for doing all three manually.
3 Publish the update

After bumping the version, publish the same way as before:

npm publish --access public

Consumers who have your package installed can update with:

npm update @royrai/ui-kit

Phase 6: Using Your Published Package

How consumers install and use it in their projects

Once published, here's how anyone (including yourself) uses the package in a new project.

1 Install the package 
npm install @royrai/ui-kit

This downloads the package from the npm registry and adds it to your project's node_modules/ folder.

2 Install peer dependencies (if needed)

Which peer dependencies you need depends on which components you import:

Component Needs MUI? Extra peer deps
RrSocialIcons No None
RrTimeWheelPicker No None
RrTimeWheelColumn No None
RoyraiCopyright No None
RoyraiCredit No None
RrCopyright No None
RrBox Yes @mui/material @emotion/react @emotion/styled
RrDataGrid Yes @mui/material @mui/x-data-grid @emotion/react @emotion/styled

For MUI-based components, install the required peer dependencies:

# Only if using RrBox or RrDataGrid:
npm install @mui/material @emotion/react @emotion/styled

# Only if using RrDataGrid:
npm install @mui/x-data-grid
3 Import components

Import individual components using the per-component paths for minimal bundle size:

// Individual imports (recommended — minimal bundle)
import { WhatsAppIcon } from "@royrai/ui-kit/controls/RrSocialIcons";
import { RrTimeWheelPicker } from "@royrai/ui-kit/controls/RrTimeWheelPicker";
import { RrDataGrid } from "@royrai/ui-kit/controls/RrDataGrid";

// Or import everything at once (larger bundle)
import { WhatsAppIcon, RrDataGrid } from "@royrai/ui-kit";
💡 Per-component imports are better

Using the specific path (e.g., @royrai/ui-kit/controls/RrSocialIcons) ensures only that component's code is included in your app's bundle. The full import (@royrai/ui-kit) pulls in everything.
